## How to use this file each run

1. Find the first unshipped item in Tier 1. Take the next **2–5 items** (not just one) that are independent of each other — batch them into a single session's work, per AGENTS.md's "bundle two or three compatible small/medium fixes into one release candidate" rule, extended here to up to 5 for same-pattern presentation-only re-skins.
2. Apply the exact proven-safe pattern from the `pages/funeral-home/summary.js` slice (2026-07-12, run 3): presentation-only re-skin onto Threshold tokens (`docs/redesign/08-visual-craft-standard.md` + `01-design-system-foundation.html`), zero changes to data-fetching/auth logic unless the page has none of its own real data to begin with (marketing/static pages), same component name and route.
3. **Run 7 caution:** if a page renders multiple conditional JSX branches through a shared helper function (e.g. `shell(content)` where `content` is a JSX literal built in each branch and passed in as an argument), do NOT use page-scoped `<style jsx>` for the shared/branch styles — babel-plugin-styled-jsx only applies its scope class to elements inside the helper's own directly-returned JSX tree, not to JSX passed in as an argument from a different return statement, so scoped selectors silently fail to match and the page renders unstyled. Use `<style jsx global>` for any styles that need to apply across content passed into a shared render-helper. Check any future page for this pattern (multi-branch state renders, e.g. loading/error/empty/success states through a shared shell) before shipping.
4. Prove the batch on a throwaway QA branch (temporarily drop `ignoreCommand` from `vercel.json`, restore it after), attempt interactive preview QA, but don't block on it if Vercel's SSO wall blocks preview access (documented, acceptable, recurring, and inconsistent limitation — confirmed again run 9, where one QA branch fetch got through and the next one on the same run did not). Note: a successful Vercel build does NOT catch styled-jsx scoping bugs like the run-7 one — it's a runtime/rendering issue, not a compile error. The live post-deploy render check (step 5) is what actually catches this, which is why it is mandatory and not skippable even when the build succeeds.
5. Deploy as one batched `[deploy][qa-approved]` release. Run the live post-deploy render check (per `release-train.md`) on **every** page in the batch, at whatever viewport this session's browser tooling actually supports — record any tooling limitation honestly rather than skipping the check. If the check finds a real rendering bug, ship a same-day hotfix commit and re-verify before considering the batch done.
6. Check off every shipped item below, with the commit SHA and date, **in the same commit** that updates `docs/agent-operating-context.md` (or a dated addendum) — see the run 8/9 notes below on why this must be one commit, not a follow-up.
7. Only once **all of Tier 1 is checked off** does a run default to starting Tier 2 (monolith splits). Tier 2 items still get split into their own reviewable sub-batches per `11-funeral-home-polish-scope.md`'s method — never a blind full-file rewrite in one pass. **Exception, run 9:** the owner directed funeral-home-critical work to take priority over finishing the Tier 1 marketing-page tail first, then confirmed the goal is the full site — so both continue in parallel across runs, funeral-home first within each run's time budget.
8. Tier 3 (internal/System Admin tooling) is lowest visual priority — functional correctness matters there, Threshold visual polish does not need to be first.
9. **Run 8 fix (verified working, run 9):** `scripts/check-agent-context.js`'s `contextTouched` check now also accepts dated addenda (`docs/agent-operating-context-YYYY-MM-DD-runN.md`) and this tracker file, not just the exact literal canonical filename. ## Pages that share chrome/design-system classes rather than owning isolated styling

`pages/funeral-home/index.js` shares `hc-*` base classes and `CalmPublicChrome` with the public homepage (`components/HomeCalm.js`). `pages/today.js` (via `components/AppCalm.js`) shares `lib/designSystem` tokens and `components/calm/CalmControls`/`CalmKit` with other consumers (checked run 8). Both are different from every page shipped so far under Tier 1/2, which each own their full styling in an isolated `<style jsx>` block. `components/SiteChrome.js` (Tier 0, above) was a third variant of this same problem class, now fixed. Re-skinning a shared-chrome page safely requires either (a) fully extending its own override layer so nothing shared visually changes, or (b) a deliberate, explicitly-scoped decision to re-skin the shared design language itself, which would affect every other consumer too. Do not treat this the same as the proven single-page pattern. Check for other pages using `CalmPublicChrome`, `lib/designSystem`, or similar shared-chrome patterns before batching them in future runs.

## Pages using a shared multi-branch render helper (styled-jsx scoping risk)

`pages/confirm.js` (run 7) used a `shell(content)` helper to wrap five different conditional JSX branches (loading/error/already/done/enter states). Its page-scoped `<style jsx>` block only had babel's scope class applied to elements inside `shell()`'s own directly-returned JSX (the `<main>`/`<div className="outer">` wrapper) — not to the branch content passed in as the `content` argument, since that JSX is authored in a separate return statement. This silently broke all styling in production (caught by the live post-deploy render check, not by the Vercel build, which succeeded normally) and required a same-day hotfix switching the scoped styles to `<style jsx global>`. Before re-skinning any future page with multiple conditional full-page states through a shared render helper, either use `<style jsx global>` for the shared styles, or restructure so the style tag lives inside the same JSX literal as the branch content it targets. **Checked run 8:** `pages/planning.js`, `pages/packet.js`, `pages/contact.js`, `pages/faq.js` are all single-return components with no shared render-helper (0 occurrences of `shell(` in each) — cleared of this risk. **Checked run 9:** `pages/funeral-home/dashboard.js`, `pages/mission.js`, `pages/story.js`, `pages/trust.js` also have 0 occurrences of `shell(` — cleared of this specific risk class. Remaining candidate to check before its own future re-skin: `components/family/FamilyTodayApp.js`.
